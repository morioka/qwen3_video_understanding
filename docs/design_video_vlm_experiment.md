# Video VLM Experiment Design

## 1. 目的

本プロジェクトの目的は、動画から単なる説明文を生成することではなく、動画から再利用可能な構造化知識を抽出することである。

対象動画:

* 保守作業動画
* 点検動画
* 障害調査動画
* 教育・手順動画

想定長さ:

* 数分～30分程度

利用モデル:

* Qwen3.6-VL
* OpenAI互換VLM
* HuggingFace上のVLM

利用音声認識:

* Whisper系

---

# 2. 基本思想

説明文を直接生成するのではなく、

動画
↓
観測・イベント抽出
↓
イベント統合
↓
必要箇所の再解析
↓
説明生成・手順書生成・チェックリスト記入

の流れを採用する。

説明文は最終成果物の一つであり、内部表現としてはイベント列やエンティティ情報を重視する。

---

# 3. 音声書き起こしの位置づけ

音声書き起こしは重要だが、主軸ではない。

保守動画では、作業者が常に丁寧に説明するとは限らない。

起こり得ること:

* 無言で重要作業をする
* 作業後にだけ説明する
* 注意点だけ話す
* 発話が作業境界と一致しない
* 発話が雑談や補足に偏る
* 映像には映るが発話されない情報がある

したがって、音声発話はチャンク境界や解釈のヒントの一つとして扱う。

「発話境界 = チャンク境界」とはしない。

---

# 4. 基本パイプライン

## Phase 1: 全体書き起こし

動画全体を書き起こす。

入力:

video.mp4

出力:

transcript_segments.json

形式:

```json
[
  {
    "start": 0.0,
    "end": 3.2,
    "text": "..."
  }
]
```

書き起こしは動画全体に対して一度だけ実施する。

チャンクごとには実施しない。

理由:

* 発話途中で切断されることを防ぐ
* 後工程で任意のチャンクに該当発話を添付できる
* チャンク境界設計と音声認識を分離できる

---

## Phase 2: チャンク境界候補の生成

チャンク境界は複数の情報源から決める。

境界候補:

* 一定時間上限
* 映像のシーン変化
* フレーム差分
* 画像特徴量の変化
* 作業対象物の変化
* 工具・部品の出入り
* 装置画面やアラーム表示の変化
* 作業者の姿勢や手元動作の変化
* 無音区間
* 発話境界
* 発話内容の手順語

手順語の例:

* まず
* 次に
* 確認します
* 交換します
* 取り外します
* 完了です
* エラーが出ました

---

## Phase 3: 初期実装のチャンク方針

最初から高度な境界推定を作り込まない。

初期版では、以下の単純で頑健な方法を採用する。

推奨初期方針:

* 30～60秒程度の時間ベースチャンク
* 数秒～10秒程度のオーバーラップ
* 各チャンクに該当時間帯の書き起こしを添付
* フレームは3～5秒間隔で抽出

この段階では、チャンク境界の精度よりも、実験を回せることを優先する。

---

## Phase 4: 改良版のチャンク方針

初期実験後、次の順に改善する。

### 改良1

PySceneDetect等によるシーン変化候補を追加する。

### 改良2

CLIP等の画像特徴量類似度により、視覚的変化点を追加する。

### 改良3

Whisperの発話境界や無音区間を補助候補として加える。

### 改良4

境界候補を統合し、以下の制約でチャンクを作る。

* 短すぎる区間は隣と結合
* 長すぎる区間は再分割
* 目標長は30～90秒
* 最大長は90～120秒程度
* 重要イベント前後は短めに切る

---

# 5. チャンクの考え方

チャンクは「発話単位」ではない。

チャンクは、

作業・観察・状態変化のまとまり

として扱う。

ただし、初期実装では完全な作業境界推定は行わない。

初期版:

時間ベース + オーバーラップ

改良版:

映像変化 + 発話情報 + 時間上限

将来版:

作業イベント境界の推定

---

# 6. チャンクへの音声添付

チャンクが決まった後、その時間範囲に重なる発話を添付する。

例:

chunk:

```json
{
  "start": 120.0,
  "end": 180.0
}
```

transcript:

```json
[
  {
    "start": 122.0,
    "end": 130.5,
    "text": "..."
  }
]
```

チャンク入力:

```json
{
  "chunk_start": 120.0,
  "chunk_end": 180.0,
  "transcript_segments": [...]
}
```

音声はチャンクを決めるための唯一の根拠ではなく、解析時に添える補助情報である。

# 7. 解析方式

## 方式A: 動画チャンク方式

入力:

* chunk_video.mp4
* 該当区間の書き起こし

特徴:

* 動作の連続性に強い
* 手を伸ばす、回す、差し込む等を捉えやすい
* モデルやAPIの動画入力対応に依存する
* 内部で何フレーム使われたか見えにくい

用途:

* 不確実区間の再解析
* 動きが重要な作業
* 重要イベント前後の精査

---

## 方式B: フレーム群方式

入力:

* チャンク内のフレーム群
* 各フレーム時刻
* 該当区間の書き起こし

特徴:

* 制御しやすい
* どの時刻の画像を使ったか追跡しやすい
* 実装が安定しやすい
* 動作の連続性は落ちる

用途:

* 初期実装
* イベント抽出
* 状態確認
* 画面表示確認
* 部品・工具・対象物の観察

---

## 推奨運用

初期実験:

方式Bを主方式とする。

改良後:

方式Bで全体解析し、不確実または重要な区間のみ方式Aで再解析する。

---

# 8. イベント抽出

各チャンクからJSONを生成する。

自由文説明より構造化出力を優先する。

形式:

```json
{
  "chunk_start": "",
  "chunk_end": "",

  "events": [
    {
      "time": "",
      "category": "",
      "actor": "",
      "action": "",
      "target": "",
      "evidence": ""
    }
  ],

  "entities": [
    {
      "id": "",
      "type": "",
      "label": "",
      "confidence": 0.0,
      "evidence": ""
    }
  ],

  "memory_update": [],
  "uncertainties": []
}
```

---

# 9. イベントカテゴリ

最低限:

* action
* state
* alarm
* speech
* observation
* unknown

例:

```json
{
  "time":"00:12:35",
  "category":"action",
  "actor":"operator_A",
  "action":"remove",
  "target":"RF Cable"
}
```

---

# 10. Entityと仮説

初期段階で不明な物体を断定しない。

例:

```json
{
  "id":"entity_17",
  "type":"object",
  "label":"unknown board-like part",
  "confidence":0.42
}
```

後続チャンクで名称が判明した場合、整合化で更新する。

例:

```json
{
  "entity_17":"RF Generator Control Board"
}
```

---

# 11. Memory

各チャンク終了時に memory_update を生成する。

次チャンクへ渡す。

例:

```json
[
  "operator_A is working on RF Generator",
  "entity_17 may be RF Generator Control Board"
]
```

ただし、memoryは確定情報だけでなく仮説も含む。

仮説には confidence を付けることが望ましい。

---

# 12. 全体整合化

チャンク解析結果をそのまま最終結果にしない。

理由:

後半で判明した情報により前半の解釈が変わるため。

例:

00:01

board

↓

00:20

RF Generator Control Board

↓

過去イベント修正

---

## 整合化対象

* エンティティ同一性
* 人物同一性
* 部品名の統一
* 作業対象の統一
* alarm/state/actionの時系列整合性
* 発話内容と映像観察の矛盾

---

# 13. Backward Refinement

整合化後、不確実な区間のみ再解析する。

対象:

* confidenceが低い
* entity未確定
* イベントが曖昧
* チェックリスト項目に関係する
* alarm/state change前後
* 映像と発話が矛盾している

再解析範囲:

±10～30秒程度

再解析では、判明済みのエンティティ名や仮説をプロンプトに入れる。

---

# 14. 最終説明生成

説明生成は最後に行う。

入力:

* 全イベント
* 全エンティティ
* memory
* 整合化結果
* 不確実事項

出力:

* 全体説明
* 時系列説明
* 重要イベント
* 映像と書き起こしの対応
* 不確実な点

---

# 15. チェックリスト利用

本システムの重要用途。

目的:

動画
↓
イベント
↓
ケース判定
↓
チェックリスト充足判定

---

例:

```json
{
  "item":"RF Cable removed",
  "status":"confirmed",
  "evidence":"00:12:35"
}
```

不明の場合:

```json
{
  "item":"Torque wrench used",
  "status":"unclear",
  "human_question":"トルクレンチを使用しましたか？"
}
```

---

# 16. 人間への質問

埋められない項目は推測しない。

状態:

* confirmed
* not_confirmed
* unclear
* not_applicable

unclearの場合のみ質問生成。
